Authentication System

OAuth Flow

Rendro uses Google OAuth 2.0 for user authentication, integrated through better-auth running inside Convex HTTP actions. The Worker proxies all auth requests to Convex.

The flow: user clicks Sign in → Worker proxies to Convex → Convex initiates Google OAuth → user authenticates → Google redirects back → Convex processes callback → session created in Convex → session cookie set → user redirected to app.

Session Management

Sessions are stored in Convex using the better-auth component's schema. Session tokens are signed JWTs stored as HttpOnly Secure cookies. The Worker's session middleware validates cookies by querying Convex's /api/auth/get-session endpoint.

Session cookies use the __Secure- prefix with SameSite=Lax, ensuring they're only sent over HTTPS and on same-site requests. The max age is configurable.

API Key Auth

CLI operations use per-organization API keys. Each key is a random 32-byte value prefixed with rendro_. The SHA-256 hash is stored in Convex; the raw key is never persisted.

API key validation is a constant-time operation — the Worker hashes the incoming key and queries Convex for a matching hash. Failed lookups return null, successful ones return the organization slug.
